# Architecture

## Overview

The app follows the standard Homey Apps SDK v3 layout with Homey Compose. All shared logic lives in `lib/`; each driver is a thin configuration layer on top of it.

```
app.ts                          Minimal App class
lib/
  types.ts                      VirtualDeviceConfig / VirtualFlowConfig type definitions
  validators.ts                 Pure helpers: parseNumber, inRange, clamp, roundTo,
                                textEquals, textContains, parseBoolean (unit-tested)
  flow.ts                       registerVirtualFlowCards(): declarative flow card
                                run-listener registration
  BaseVirtualDevice.ts          Shared device behavior
  BaseVirtualDriver.ts          Shared driver behavior (flow registration + pairing)
drivers/virtual-<type>/
  driver.compose.json           Name (10 languages), class, capabilities, pairing
  driver.flow.compose.json      Flow cards with translations
  driver.settings.compose.json  Device settings (only number and string)
  driver.ts                     extends BaseVirtualDriver — flowConfig object
  device.ts                     extends BaseVirtualDevice — config object
.homeycompose/
  app.json                      App manifest source
  capabilities/                 Custom capabilities: virtual_number, virtual_text
locales/<lang>.json             Runtime strings (user-facing errors)
```

The root `app.json` is generated by the Homey CLI from the compose sources. Never edit it directly.

## Value flow

Every value change goes through one path in `BaseVirtualDevice`:

1. **Input** — either the capability listener (Homey UI, API, built-in flow cards) or `setValue()` (this app's flow actions).
2. **`normalizeValue()`** — validates the type and applies constraints. Throws a localized `Error` for invalid input (shown to the user by Homey). Numeric configs can declare `clampMin`/`clampMax` (battery & humidity 0–100, luminance/pressure/CO2/energy ≥ 0, dim 0–1) which the base class applies. Subclasses override it only for special cases: `VirtualNumberDevice` rounds to the configured decimals and clamps to the settings range when enabled.
3. **`writeValue()`** — compares with the current value; when changed it persists via `setCapabilityValue()` and calls `handleChange()`.
4. **`handleChange()`** — fires the driver's `*_changed` trigger with `value` and `previous` tokens, fires the optional `trueTriggerId`/`falseTriggerId` cards for booleans (button turned on/off, contact opened/closed, motion started/ended), then calls the `onValueChanged()` hook (the dimmer uses it for percentage tokens and onoff sync).

Flow card run listeners are registered once per driver in `BaseVirtualDriver.onInit()` via `registerVirtualFlowCards()`, driven by the driver's `flowConfig`. Trigger cards need no run listener; they are fired from the device.

## Persistence

Homey persists capability values natively, so values survive app restarts and Homey reboots. On `onInit()` the device checks the stored value and only applies the safe default (`config.defaultValue`) when no valid value exists yet (i.e. right after pairing).

## Pairing (custom view with icon picker)

Every driver uses the shared custom pairing view `pair/start.html` (see ADR-0004): a name field prefilled with the localized driver name plus a grid of 27 icons. The view calls `Homey.createDevice({ name, data: { id }, icon })` directly; `BaseVirtualDriver.onPair()` only serves the `getPairInfo` handler (localized name + `crypto.randomUUID()`).

Single source of truth for pairing assets:

- `assets/device-icons/*.svg` — the icon library
- `pairing/start.template.html` — the view (texts via `locales/*.json` `pair.*` keys)
- `scripts/sync-pairing.js` (`npm run sync:pairing`) copies icons into every `drivers/<id>/assets/icons/` and generates every `drivers/<id>/pair/start.html` (icon grid inlined, per-driver default icon preselected). Generated files are excluded from Prettier; run the script after changing icons or the template.

## Adding a new virtual type

1. `drivers/virtual-<type>/driver.compose.json` — pick a standard Homey capability when one exists (preferred: `measure_humidity`, `measure_pressure`, `alarm_contact`, `dim`, …); otherwise add a custom capability under `.homeycompose/capabilities/`.
2. `driver.flow.compose.json` — copy the card pattern of the closest existing driver (numeric → temperature, text → string, boolean → boolean/button) and translate all 10 languages. Cards with args need `titleFormatted`.
3. `driver.ts` — `flowConfig` names the condition/action card ids; `lib/flow.ts` already knows how to run in-range/equals/contains/set/toggle listeners.
4. `device.ts` — `config` names the capability, value type, default and changed-trigger id. Override `normalizeValue()` only for extra constraints.
5. `npm test` enforces translation completeness and `titleFormatted` coverage; `npm run validate` checks the manifest.

## Error handling & logging

- Invalid flow-action input throws localized errors (from `locales/*.json`) so the user sees why the card failed.
- Trigger failures are caught and logged with `this.error()` — they never break the value update.
- Logging is limited to one line per device/driver init and error paths; no per-value-change spam.
