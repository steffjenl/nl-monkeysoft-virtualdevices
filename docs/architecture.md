# Architecture

## Overview

The app follows the standard Homey Apps SDK v3 layout with Homey Compose. All shared logic lives in `lib/`; each driver is a thin configuration layer on top of it.

```
app.ts                          Minimal App class
lib/
  types.ts                      VirtualDeviceConfig / VirtualFlowConfig type definitions
  validators.ts                 Pure helpers: parseNumber, inRange, clamp, roundTo,
                                textEquals, textContains, parseBoolean (unit-tested)
  flow.ts                       registerVirtualFlowCards(): declarative flow card
                                run-listener registration
  BaseVirtualDevice.ts          Shared device behavior
  BaseVirtualDriver.ts          Shared driver behavior (flow registration + pairing)
drivers/virtual-<type>/
  driver.compose.json           Name (10 languages), class, capabilities, pairing
  driver.flow.compose.json      Flow cards with translations
  driver.settings.compose.json  Device settings (only number and string)
  driver.ts                     extends BaseVirtualDriver ‚Äî flowConfig object
  device.ts                     extends BaseVirtualDevice ‚Äî config object
.homeycompose/
  app.json                      App manifest source
  capabilities/                 Custom capabilities: virtual_number, virtual_text
locales/<lang>.json             Runtime strings (user-facing errors)
```

The root `app.json` is generated by the Homey CLI from the compose sources. Never edit it directly.

## Value flow

Every value change goes through one path in `BaseVirtualDevice`:

1. **Input** ‚Äî either the capability listener (Homey UI, API, built-in flow cards) or `setValue()` (this app's flow actions).
2. **`normalizeValue()`** ‚Äî validates the type and applies constraints. Throws a localized `Error` for invalid input (shown to the user by Homey). Subclasses override it:
   - `VirtualNumberDevice` rounds to the configured decimals and clamps to min/max when the range setting is enabled.
   - `VirtualBatteryDevice` clamps to 0‚Äì100.
3. **`writeValue()`** ‚Äî compares with the current value; when changed it persists via `setCapabilityValue()` and calls `handleChange()`.
4. **`handleChange()`** ‚Äî fires the driver's `*_changed` trigger with `value` and `previous` tokens, then calls the `onValueChanged()` hook (the button uses it for its turned on / turned off / toggled triggers).

Flow card run listeners are registered once per driver in `BaseVirtualDriver.onInit()` via `registerVirtualFlowCards()`, driven by the driver's `flowConfig`. Trigger cards need no run listener; they are fired from the device.

## Persistence

Homey persists capability values natively, so values survive app restarts and Homey reboots. On `onInit()` the device checks the stored value and only applies the safe default (`config.defaultValue`) when no valid value exists yet (i.e. right after pairing).

## Pairing

`BaseVirtualDriver.onPairListDevices()` returns exactly one new device per pairing session, named after the driver in the user's language, with a `crypto.randomUUID()` as device id. Users rename the device in the Homey UI.

## Adding a new virtual type

1. `drivers/virtual-<type>/driver.compose.json` ‚Äî pick a standard Homey capability when one exists (preferred: `measure_humidity`, `measure_pressure`, `alarm_contact`, `dim`, ‚Ä¶); otherwise add a custom capability under `.homeycompose/capabilities/`.
2. `driver.flow.compose.json` ‚Äî copy the card pattern of the closest existing driver (numeric ‚Üí temperature, text ‚Üí string, boolean ‚Üí boolean/button) and translate all 10 languages. Cards with args need `titleFormatted`.
3. `driver.ts` ‚Äî `flowConfig` names the condition/action card ids; `lib/flow.ts` already knows how to run in-range/equals/contains/set/toggle listeners.
4. `device.ts` ‚Äî `config` names the capability, value type, default and changed-trigger id. Override `normalizeValue()` only for extra constraints.
5. `npm test` enforces translation completeness and `titleFormatted` coverage; `npm run validate` checks the manifest.

## Error handling & logging

- Invalid flow-action input throws localized errors (from `locales/*.json`) so the user sees why the card failed.
- Trigger failures are caught and logged with `this.error()` ‚Äî they never break the value update.
- Logging is limited to one line per device/driver init and error paths; no per-value-change spam.
